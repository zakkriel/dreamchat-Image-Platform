# ADR-011 — Use S3-compatible object storage for image bytes

## Status

Accepted for initial implementation.

## Context

The platform produces and serves image bytes at three tiers (thumbnail, preview, final). It also needs structured metadata about each asset (variant, version, identity, prompt hash, provider, cost). These have different access patterns: bytes are large, served via signed URLs, served-many-read-once; metadata is small, queried frequently, cross-joined.

Where the bytes live affects: cost per GB stored, cost per GB served, write/read latency, dev experience, and vendor portability.

## Decision

Image bytes live in **S3-compatible object storage** (AWS S3, R2, GCS via S3 gateway, MinIO in local/dev). All metadata lives in Postgres (ADR-012). Object paths follow PRD 05's structured layout (`/{tenant}/{world}/{entity_type}/{entity_id}/{vid_version}/{asset_role}/{asset_id}/{tier}.webp`), but the path is *not* the source of truth — `visual_assets.{low_res,high_res,thumbnail}_url` in Postgres is.

## Alternatives considered

- **Local filesystem.** Fine in single-node dev. Doesn't survive replicas, autoscale, or container restarts. We'd need an NFS or sidecar pattern that ends up looking like a worse S3.
- **Postgres bytea / large objects.** One system to operate. Blows up DB size, slows backups, makes streaming reads awkward, and conflates byte storage with transactional metadata. Rejected immediately for any realistic volume.
- **Cloudflare Images / image-CDN-as-storage.** Bundles storage + CDN + transforms. Lower ops burden. But ties storage to a single vendor and makes moving providers (or going self-hosted) hard. Worth considering as a *front* later, not as the canonical store.
- **IPFS / content-addressed storage.** Content addressing is appealing (matches `prompt_hash`). Operationally heavy for a startup, no managed providers in our deploy environment.
- **Direct upload from provider to S3 (signed PUT URL).** Saves a round trip. Most providers don't support it; complicates the worker logic and the bookkeeping when a provider does support it but stalls mid-upload.

## Tradeoffs

- **+** Cheap and scalable; storage cost is a small fraction of generation cost.
- **+** MinIO gives local-dev parity with one-line docker-compose.
- **+** Decouples storage from compute — workers can be restarted/scaled without touching bytes.
- **+** Signed URLs let the web app fetch bytes directly without proxying through the API.
- **−** Two systems to keep consistent (object store + Postgres). Orphan-file risk if a metadata write fails after a successful upload (mitigated by upload-then-commit pattern).
- **−** Signed URLs add complexity vs. direct serve (TTL choice, rotation on token revocation, CORS for browser fetch).
- **−** Deriving thumbnails/previews from a single final asset is a separate pipeline step (or generated by the provider).

## Consequences

- Storage interface in `internal/storage` exposes `PutObject`, `GetObject`, `PresignURL`. Adapters: AWS SDK (`aws-sdk-go-v2`) or MinIO Go client.
- `visual_assets.{thumbnail_url, low_res_url, high_res_url}` are the authoritative pointers.
- Upload sequence: upload bytes → derive thumbnail+preview (if not provider-supplied) → atomically write `visual_assets` row referencing all three URLs.
- Orphan file cleanup runs as a future maintenance job — out of MVP scope.

## Revisit when

- A measurable fraction of cost shifts from generation to storage/bandwidth (rare given generation costs — but possible at scale, then revisit CDN + image-CDN options).
- We hit object-store rate limits for a hot world's assets (sharding by tenant or world prefix is already supported by the path scheme).
- A new asset type (video, animation — explicitly out of MVP scope) needs a different storage shape.

---

## Confidence to Implement

**Score: 95/100 — Very High**

`aws-sdk-go-v2` (or `minio-go`) plus a thin storage interface (`PutObject`, `GetObject`, `PresignURL`) covers everything needed. The key layout in PRD 05 is sensible. MinIO works locally and in CI. The only thinking left is signed-URL TTL choice and whether to fan out three derivatives (thumbnail/preview/final) at write time or generate on demand — both are easy.
